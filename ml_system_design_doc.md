# ML System Design Doc — AI HR: верификация компетенций и формирование требований к вакансии

## 1. Цели и предпосылки

### 1.1. Зачем идём в разработку продукта?

#### Бизнес-цель
Снизить трудозатраты и неопределённость в найме за счёт стандартизации требований к вакансии и воспроизводимой проверки компетенций кандидатов.

#### Какая проблема рынка закрывается (практическая направленность)
- Вакансии часто содержат размытые формулировки (“уверенный Python”), которые разные участники процесса трактуют по-разному.
- Результат: высокий шум на входе, дорогой скрининг, много неподходящих кандидатов на интервью, время инженеров уходит на повторяющуюся проверку базы (Python/SQL/ML основы).
- Для кандидатов проблема обратная: непонятно, какие конкретные знания и на каком уровне нужны, чтобы соответствовать вакансии/рынку.

#### Почему станет лучше от использования ML
Сейчас требования и оценка кандидатов делаются вручную/по общим словам (“уверенный Python”), из-за чего:
- кандидатов сложно сравнивать между собой;
- требования к вакансии получаются разными у разных людей;
- процесс дорогой по времени (перечитывание резюме/портфолио, проверка на собеседованиях).

ML/LLM используется только там, где это оправдано:
- нормализация навыков к единому справочнику (онтология + синонимы);
- извлечение требований из текста вакансии;
- агрегирование сигналов по навыкам и расчёт уровня владения.

Без ML можно сделать только словарь/регулярные выражения и простые правила, но они плохо работают на:
- синонимах и контексте (“Spark” как фреймворк vs “spark” как слово);
- неявных требованиях (“оптимизация запросов”, “витрины данных”) — это про SQL/DB, но слово SQL может не встречаться;
- разной формулировке одного уровня (“production experience”, “опыт внедрения”, “поддержка пайплайна”).

ML + онтология позволяют:
- нормализовать навыки и связать формулировки с каноническими компетенциями;
- оценивать требуемый уровень по контексту обязанностей и задач;
- сопоставлять вакансии и кандидатов в общем пространстве (вектор компетенций).

#### Почему это нужно решать именно системой, а не вручную
Ручная нормализация требований и оценка кандидата не масштабируется в потоке. Система задаёт единые правила описания компетенций и делает сравнение кандидатов воспроизводимым.

#### Что будем считать успехом итерации с точки зрения бизнеса (Product Owner)
Итерация 1 успешна, если система в пилоте:

1) **Не пропускает критичные требования вакансии.**  
   - **Нельзя пропускать:** базовые компетенции (например, для DS: Python, SQL, базовый ML, основы статистики/экспериментов). Если система их не выделила — она построит неверный профиль и матчинг будет бессмысленным.  
   - **Можно пропускать:** редкие/вариативные компетенции (например, конкретный облачный провайдер или BI-инструмент), если они явно не указаны как обязательные.  
   - **Критерий:** по пилотному набору вакансий система корректно выделяет все обязательные компетенции в **9 из 10** вакансий. Если ниже — пилот считается провальным.

2) **Не предлагает работодателю заведомо неподходящих кандидатов.**  
   - Заведомо неподходящий кандидат — тот, кто не закрывает обязательные требования (например, SQL на нужном уровне), но ранжируется высоко в топе.  
   - **Критерий:** в топе допускается **не более 2** кандидатов с провалом по обязательным компетенциям.

3) **Экономит время формирования требований и шорт-листа.**  
   - **Критерий:** подготовка профиля вакансии с правками занимает **< 20 минут** (вместо ручного 60–120), а первичный shortlist формируется быстрее **как минимум в 2 раза** (замер в пилоте “как было” vs “как стало”).

### 1.2. Бизнес-требования и ограничения

#### Краткое описание БТ (Product Owner)
Основные бизнес-требования:

1) **Автоматически формировать требования к вакансии из текста** (вакансия/описание роли/публичные задания).  
   Выход — список компетенций с:
   - требуемым уровнем (0–5),
   - важностью (вес),
   - типом (обязательные / желательные),
   - коротким объяснением требования для прозрачности.

2) **Оценивать компетенции кандидата** на основе его артефактов в системе обучения/контеста и/или задач/проектов.  
   Выход — уровень владения по навыкам + “доказательства” (какие задания/темы закрыты, где ошибки).

3) **Ранжировать кандидатов под вакансию** и показывать:
   - итоговый fit-score,
   - провалы must-have,
   - skill gap (что подтянуть),
   - объяснение результата.

4) **Настройка в интерфейсе** (обязательное бизнес-требование):  
   пользователь должен уметь руками:
   - поменять must/nice,
   - изменить веса,
   - скорректировать требуемый уровень,
   - сохранить профиль вакансии как шаблон.

#### Бизнес-ограничения (Product Owner)
- Источники требований: только общедоступные (GitHub-репозитории с заданиями/вопросами, публичные вакансии, публичные материалы).
- Результат должен быть объясним: “почему такой уровень/вес”.
- Ошибки разного класса: ошибка по must-have критичнее, чем ошибка по nice-to-have (это отражается в логике ранжирования).
- Хранение: в пилоте минимизируем персональные данные — достаточно `candidate_id` + результаты задач/темы/оценки (без лишних PII).

#### Описание бизнес-процесса пилота (Product Owner)
1) Нанимающий/HR вставляет текст вакансии или выбирает профиль “DS junior/middle”.
2) Система строит “черновик требований”: навыки/уровни/веса/must-have.
3) Пользователь быстро правит и сохраняет профиль.
4) Пользователь выбирает пул кандидатов → получает топ + объяснения и skill gap.
5) Эксперт подтверждает качество top-выдачи и фиксирует ошибки для следующей итерации.

#### Успешный пилот (Product Owner)
Пилот успешен, если выполнено gate-условие (не пропускаем must-have) и достигнут эффект по времени/качеству top-выдачи. Если gate не выполнен — пилот неуспешен независимо от остальных пунктов (система не вызывает доверия).

### 1.3. Что входит в скоуп итерации, что не входит

#### Входит
- формирование требований к вакансии из текста + редактирование в UI;
- база публичных задач/вопросов → связь “задание → проверяемые навыки/уровень”;
- оценка кандидата по результатам задач/контеста (интеграция с “Группой 1” через вектор прогресса);
- матчинг и ранжирование + skill gap + объяснения.

#### Не входит (в MVP)
- полноценная обработка резюме и социального следа кандидата;
- оценка soft skills;
- покрытие всех ИТ-ролей (в MVP только DS/ML).

### 1.4. Предпосылки решения

- Есть онтология компетенций (skill dictionary): канонические навыки + синонимы + категории.
- Для ключевых навыков есть рубрики уровней 0–5 (как SQL-пример); иначе “уровень” будет невалидным.
- Есть/будет база публичных заданий/вопросов, где каждое задание связано с навыками (multi-label) и минимальным уровнем.
- Контест/учебный сервис (Группа 1) отдаёт структурированный результат (что решено/ошибки/темы), чтобы строить evidence-профиль кандидата.
